# Variables, Precedence, and Project Logic

Этот документ объясняет, как в проекте работают переменные, слои конфигурации и расширение ролей.

## 1. Главное правило

Порядок чтения файлов и приоритет переменных в Ansible не одно и то же.

- Файлы могут загружаться в одном порядке.
- Итоговое значение переменной определяется уровнем precedence.

## 2. Практический порядок приоритета (снизу вверх)

1. `roles/<role>/defaults/main.yml`
2. `inventories/<env>/group_vars/all/*.yml`
3. `inventories/<env>/group_vars/<group>/*.yml`
4. `inventories/<env>/host_vars/<host>/*.yml`
5. `vars` внутри playbook
6. `-e/--extra-vars` (обычно самый высокий приоритет)

`ansible-vault` не повышает приоритет, он только шифрует значения.

## 3. Как это применено в этом репозитории

1. `inventories/<env>/hosts.yml` определяет хосты и группы.
2. `group_vars` задает базовую политику для окружения и групп.
3. `host_vars/<host>/vault.yml` хранит host-specific секреты и переопределения.
4. `playbooks/prod/*.yml` определяют порядок ролей.
5. Внутри ролей задачи выполняются сверху вниз из `tasks/main.yml`.

## 4. Роль `hash_behaviour = merge` в этом проекте

В `ansible.cfg` включен `hash_behaviour = merge`.

Это значит:

1. Для словарей проект использует модель "база + точечные переопределения".
2. Изменения в `group_vars/<group>` и `host_vars/<host>` дополняют базу из `group_vars/all`.
3. Списки обычно не merge-ятся автоматически и перезаписываются целиком.

Если сменить `merge` на `replace`, текущая логика слоистой конфигурации может работать иначе.

## 5. Почему здесь нет префиксов `00_`, `10_`, `20_`

Для стандартной схемы `group_vars`/`host_vars` префиксы чаще мешают:

- они не меняют precedence;
- добавляют шум и усложняют навигацию.

Префиксы полезны только если вы вручную строите слоистую загрузку через `include_vars`/`vars_files`.

## 6. Где менять проект правильно

1. Включение или выключение функций: `key_components.*.enabled` в `group_vars`.
2. Каталог пакетов и репозиториев: `key_catalog` в `group_vars/all/main.yml`.
3. Секреты и host-specific значения: `host_vars/<host>/vault.yml`.
4. Новая логика: новая роль или новая задача внутри роли.

## 7. Безопасность секретов

1. Рабочие `vault.yml` храните только в зашифрованном виде.
2. Пароль vault не храните в Git.
3. Для публикации оставляйте шаблоны (`template.yml`) и безопасные заглушки.
